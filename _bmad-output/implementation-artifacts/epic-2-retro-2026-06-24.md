## 2. What Went Well (Strengths)

- **The 15-minute de-risk spike became the velocity engine.** Stories 2.2, 2.3, 2.4, and 2.5 each
  ran a throwaway spike on the installed `sqlite-vec` v0.1.9 **before** writing code. This evolved
  Epic 1's "decisions locked up front" into empirical verification of the actual wheel — and
  implementation then ran clean (TDD red→green, little logic iteration).
- **Three wrong planning-doc assumptions caught before they became bugs** — all by spikes/analysis,
  not review:
  1. **`card_id` TEXT-PK** (2.2) — every planning doc said *"rowid = card_id"*, but the real
     `cards.id` is a Scryfall UUID string; a SQLite rowid cannot hold a UUID. Declared
     `card_id TEXT PRIMARY KEY` instead.
  2. **`INSERT OR REPLACE` is silently broken on `vec0`** (2.3) — re-inserting a `card_id` raises
     `UNIQUE constraint failed` *even with* `OR REPLACE`. Resolved with DELETE-then-INSERT.
  3. **Footprint reconciliation** (2.3) — NFR4's "~92 MB" is the 60k projection; the real DB has
     **38,232** cards (~67 MB actual). Avoided a false-failing hard threshold.
- **Compounding, frozen ports.** `hybrid_search` was built embed-agnostic in 2.4 *specifically so*
  2.5 could reuse it verbatim — 2.5 added only `get_card_vector` + one `exclude_oracle_id` arg.
  The "What this story IS — and is NOT" + anti-patterns sections held across all six stories.
- **Previous Story Intelligence chain.** Each story carried prior lessons forward and pre-empted
  recurring review findings (relative-path, `Example:` docstrings, `tmp_path`, `factory.close()`,
  positional row indexing, SQL built from schema constants, all user values bound as parameters).
- **Real-corpus smokes validated NFR1** (40–51 ms on the 38k index) and honest semantic ranking
  ("flying red dragon …" → red dragons nearest-first; `find_similar(Glorybringer)` → other
  aggressive red dragons, seed oracle absent, 47.8 ms).
- **The Epic-1 prep gate paid off** — the nullability coercion validators were in place before
  2.3's builder read all 38k cards, so the predicted blocker never materialized.

## 3. Challenges & Growth Areas (recurring patterns)

- **Recurring deferrals — Epic 1's takeaway #3, not applied.** Two cheap items were deferred
  repeatedly across reviews:
  - **`_FakeEmbedder` duplication** — 2.4 review flagged 3 copies; 2.5 added a **4th**. Tracked in
    `deferred-work.md` since 2.4, never consolidated.
  - **`limit > over_fetch_k` silent starvation** — flagged in 2.4, deferred; re-flagged in 2.5 with
    an extra wrinkle (a seed with many printings consumes KNN slots before exclusion). No
    upper-bound validation on `limit`.
  - **Root cause:** work that is individually cheap and individually deferrable never crosses the
    threshold to break scope discipline. The deferred ledger *tracks* well; nothing *promotes* a
    tracked item to "do it now" on the Nth repeat. Epic 1 named the trigger; it was never wired in.
- **Infra exceptions surface as `isError=True`.** `OperationalError` (e.g. index not built),
  `RuntimeError` (ONNX), and `JSONDecodeError` propagate uncaught from the sync tools. Consistent
  with the Epic-1 async pattern — but Epic 3's skills sit *on top* of these tools, so a raw error
  there is a worse UX than a graceful `status`.
- **LIKE wildcard injection** — `%`/`_` in seed/card names un-escaped; pre-existing from Epic 1
  (`card_lookup`, `CardRepository`), now inherited by `find_similar`. Accepted risk, mirrors the
  repo, but now in a third place.

## 4. The RAG Eval — honest scope (a strength *and* a limitation to carry)

Story 2.6's eval asserts an aggregate top-K hit-rate against a **hermetic curated 20-card fixture**
(observed 12/12 = 1.0; `TARGET_HIT_RATE = 0.9`). The optional smoke against the **real 38,232-card
index** hit only **2/12** — not a recall defect, but because over 38k cards there are dozens of
equally-valid burn spells / board wipes / draw spells, so "this *specific* expected card in top-5"
is a far harder, ambiguous task than over a curated pool with one unambiguous best match each.

- **Implication:** the eval guards **regression / plumbing**, not **production specific-card recall**.
  Its character on the real corpus is **breadth over precision** (semantically on-target).
- This is a **design input for Epic 3**, not a course-correction (see §6).

## 9. Key Takeaways

1. **Empirical de-risk beats documented decisions** — verifying the installed `sqlite-vec` wheel
   with a 15-min spike caught three wrong planning assumptions before code. Carry it into Epic 3 as
   "dry-run the orchestration on the real index first."

   across reviews because nothing forced the Nth repeat to "do it now." The Targeted Gate is that
   promotion; wire the trigger into review triage going forward.
3. **Know your recall character before building judgment on it** — the eval guards regression, not
   real-corpus precision; Epic 3 skills should present ranked candidates with reasons.
