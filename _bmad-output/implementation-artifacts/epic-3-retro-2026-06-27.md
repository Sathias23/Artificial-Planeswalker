## 2. What Went Well (Strengths)

- **The "previous-story intelligence" chain produced a visible learning curve.** Review patch counts fell **16 → 6 → 1** across 3.1→3.4. Story 3.1 absorbed 16 contract-fidelity patches as the first skill; 3.2/3.3/3.4 carried 3.1's review findings forward as a *pre-emptive checklist* (off-convention status enums like `lookup_card_by_name`→`found`, format-precedence-not-silent-default, `page_size` silent-clamp vs `limit` hard-reject) and stopped re-introducing them. The Epic-2 "compounding frozen ports" idea applied to *prose*: each skill was modeled on the prior sibling's `SKILL.md`.
- **Every capability skill has a real, source-grounded reason to exist.** Each one's value-add is reasoning *past* a tool's hard-coded limitation, and each correctly pinned what its tool is blind to: `detect_synergies` (only tribal + 12 keywords + 3 mechanic combos), `analyze_mana_curve` (format- and archetype-blind, hard-codes ~40% lands), `validate_deck` (constructed-60 hard-coded; singleton / 100-card / Limited-40 all unchecked).
- **The live dry-run caught tool behavior no doc-reading would have.** The foundational one: the **candidate-generator pattern** — semantic search ranks by *topical proximity, not logical conjunction* (the best "removal that also reanimates" card ranked **14th** in `TOOL_PERFORMANCE_REPORT.md`). That single finding became the spine of all four skills (over-fetch → intersection-filter; `distance` is within-call-only; present ranked candidates with reasons, never "the one right card").
- **Dry-runs surfaced sneaky, now-documented failure modes:** the `//` double-faced-card junk-"tribe" false-positive in `detect_synergies` (3.2), and the **lowercase-format trap** in 3.4 — `validate_deck` does `.strip()` but not `.lower()`, so a capital-S `"Standard"` flags *every* card illegal (proven live: 39 violations including the basic lands).
- **Near-zero deferrals — the Epic-2 correction held.** Patches were applied on the spot, not kicked down the road. Only 1 deferral across the epic (a pre-existing `validate_deck` orphaned-card-join edge, not skill debt). Epic 1's takeaway #3 (which Epic 2 *failed*) is now sustained.

## 3. Challenges & Growth Areas

- **3.1 paid the first-of-a-kind tax (16 patches).** All contract-fidelity, no AC blockers — but it tells us the verified tool-contract table should be **extracted once and shared**, not re-typed per skill. Four skills now carry near-identical contract tables that can drift independently. → **D1.**
- **⭐ The product's judgment is client-locked to Claude Code (the headline).** The skills are what make the raw tools *good* — they paper over `validate_deck` being constructed-60-only, the topical-vs-conjunction recall character, the format-key trap, the `//` parser bug. But Phase 2 (Letta agent) and Phase 3 (Electron UI) are **different MCP clients** that inherit the raw tools, **not the judgment**. A subset of that judgment is actually *compensating for tool defects that should be fixed at the source* so every client benefits. **Brad: "we need to tackle this, for sure."** → **SD1 / SD2 / Pre-Phase-2 Gate.**
- **One deferral (3.4):** `validate_deck` counts orphaned card-joins toward `mainboard_count` but skips them in copy/legality checks — a "legal" result can hide phantom cards. Pre-existing tool/data edge, correctly deferred.

## 5. Next Direction — Phase 1 Complete; Phase 2 = Import/Export

There is **no Epic 4 defined.** Epic 3 was the last epic of Phase 1; the epic breakdown routes Phase 2 (Letta) and Phase 3 (Electron UI) through their own spec → plan → implementation cycles.

**Brad's call: Phase 2 is the MTG Arena deck import/export feature** (2026-06-27 research: feasible, ~4-day, decisions scoped — `arena_printing` map, deck-zone gap, set-code map).

**The connection that makes the two findings one:** Phase 2 being import/export is the place to *fix* the portability problem instead of repeating it. Build import/export as **MCP tools with logic in `src/logic` (capability-as-tool, not capability-as-skill)** — the precedent that answers the client-locked-judgment concern.

### 🔎 Significant design input (no epic-rewrite required — there is no defined epic to rewrite)
The portability finding **must enter the Phase 2 spec on day one.** A chunk of skill judgment is genuine LLM reasoning (coaching tone, ranked-candidate intersection-filter, sideboard strategy) that legitimately stays a skill; another chunk is the skills papering over **tool defects** that should migrate to `src/` so all clients benefit:

| Skill-encoded workaround | `src/` fix candidate (portable) | Cost |
|---|---|---|
| 3.4 lowercase-format trap (every card flags illegal) | `validate_deck` `.lower()` + valid-key guard | **Cheap — kills a catastrophic silent failure for all clients** |
| 3.4 `validate_deck` constructed-60-only (no singleton/100/Limited-40) | format-aware structural rules (the D-1.6b limitation) | Medium |
| 3.2 `detect_synergies` `//` junk "tribes" | parser fix in `_extract_creature_types` | Cheap |
| 3.3 `analyze_mana_curve` format/archetype-blind | partial — some is genuinely heuristic judgment | Larger |

## 8. Key Takeaways

1. **Carrying prior review findings forward compounds — even for prose.** The previous-story-intelligence chain drove review patches 16→6→1; treat each shipped artifact's review as the next one's pre-emptive checklist.
2. **Judgment that lives only in a skill is client-locked.** What makes the tools good is trapped in `.claude/skills/`; a subset of it is compensating for tool defects that should be fixed at the source. Phase 2 is the moment to set the capability-as-tool precedent.
3. **Dry-run before you encode.** Live tool behavior (topical-not-conjunction recall, the `//` parser bug, the lowercase-format trap) is invisible from the docs and decisive for the skill — verify it first. Sustained from Epic 2, and it paid off again.
